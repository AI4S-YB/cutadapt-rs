Codex speed-up session history
==============================

Date
----

- 2026-04-10


Context
-------

This session focused on performance analysis and optimization of `cutadapt-rs`,
starting from the benchmark reported in `README.md` for commit `0ad2e08`.

Initial public benchmark:

- Dataset: 9,880,559 paired-end reads
- Read length: 150 bp
- Platform: Illumina
- Scenario: standard TruSeq adapter trimming
- Python cutadapt 5.2: `1m 07s`
- Rust cutadapt-rs: `1m 47s`
- Output statistics were numerically identical


Phase 1: Bottleneck analysis only
---------------------------------

The user first asked for optimization ideas only, without code changes.

Main conclusions:

1. The top structural issue was not the adapter matching core itself, but the
   surrounding pipeline.

2. The most important bottlenecks were identified as:
   - full-input buffering via `read_all()`
   - single-threaded gzip output
   - hot-path cloning, repeated allocation, and repeated modifier construction

3. Specific baseline code locations discussed:
   - `crates/cutadapt/src/main.rs`
     - `read_all()` in the main pipeline
     - per-read construction of trimmers/cutters
     - per-read `write_record()` calls
   - `crates/cutadapt-core/src/files.rs`
     - gzip writer based on `flate2::write::GzEncoder`
     - parsing paths that allocate and copy eagerly
   - `crates/cutadapt-core/src/info.rs`
     - `ModificationInfo::new(record.clone())`
   - `crates/cutadapt-core/src/record.rs`
     - `slice()` creating new owned strings
   - `crates/cutadapt-core/src/adapters/matches.rs`
     - match objects copying sequence and adapter metadata

Recommended optimization order from the analysis phase:

1. Fix the gzip/write path first.
2. Replace `read_all()` with streaming or chunked processing.
3. Remove hot-path clones and repeated object construction.
4. Only after that, consider deeper matcher or multi-core processing work.


Phase 2: Optimization in a new worktree
---------------------------------------

The user then requested implementation in a new worktree.

Worktree details:

- Path: `/tmp/cutadapt-rs-speedopt`
- Branch: `perf/speed-opt-20260410`

Main code changes made there:

1. Streaming / chunked pipeline
   - Replaced whole-file `read_all()` style execution with batched processing.
   - Allowed read/process/write overlap instead of materializing all reads first.

2. Output path restructuring
   - Switched to batch serialization into buffers before write-out.
   - Reduced fine-grained per-record write overhead.

3. Modifier reuse
   - Moved `UnconditionalCutter`, `QualityTrimmer`, `NextseqQualityTrimmer`,
     `NEndTrimmer`, and `Shortener` construction out of the hot per-read loop.

4. Reduced hot-path cloning
   - Added a lightweight `ModificationInfo` constructor that does not keep a
     cloned copy of the original read when unnecessary.

5. Parser allocation cleanup
   - Introduced a `SequenceRecord::from_parts(...)` path so parsed strings could
     be moved instead of copied again.

6. Parallel gzip writer
   - Added a parallel gzip writing stage.
   - Wired `--cores` into the compression path.
   - Changed the CLI default so `--cores=0` means auto and is the default path.

7. Test stability cleanup
   - Added a mutex around adapter-name-counter tests to avoid cross-test races.

Files changed in the worktree:

- `README.md`
- `crates/cutadapt/src/main.rs`
- `crates/cutadapt-core/src/files.rs`
- `crates/cutadapt-core/src/info.rs`
- `crates/cutadapt-core/src/record.rs`
- `crates/cutadapt-core/src/adapters/mod.rs`


Phase 3: Benchmark results after optimization
---------------------------------------------

Measured on the same workload:

Baseline:

- Python cutadapt 5.2: `1m 07s`
- Rust baseline at `0ad2e08`: `1m 47s`

Intermediate result after the first optimization round:

- `/dev/null`: `28.93s`
- gzipped output: `1m01.46s`
- Peak RSS dropped from roughly `8.5 GB` to roughly `42-51 MB`

Final result after parallel gzip and the completed restructuring:

- gzipped output: `30.59s`
- `/dev/null`: `28.93s`
- Peak RSS: about `135 MB`

Interpretation:

- The pipeline and write/compression structure were indeed the major issues.
- The final runtime was no longer dominated by the same factors as the baseline.
- This validated the earlier conclusion that directly micro-optimizing the
  matcher first would have been the wrong priority.

Phase 7: Continued optimization on main
---------------------------------------

After the earlier restructuring merge, optimization continued directly on
`main` rather than in the temporary worktree.

The next focus was the remaining gap in multi-threaded runs.

New observations from benchmark discussion:

- single-thread performance had become much closer to Python
- the larger remaining gap showed up mostly in multi-core runs
- this suggested the next bottlenecks were no longer the original
  `read_all()` / single-thread gzip issues, but rather:
  - read-processing parallelism
  - pipeline scheduling
  - compression/decompression implementation cost
  - thread-budget interactions between processing and gzip

Main implementation work done on `main`:

1. Added persistent worker-based read-processing parallelism
   - moved away from single-threaded batch processing
   - introduced batch jobs/results with ordered merge on the main thread
   - preserved output order and summary counters

2. Reworked scheduling and memory behavior
   - reduced in-flight batch depth
   - changed worker dispatch to a shared queue model
   - moved per-batch FASTQ serialization into worker threads so the main
     thread mostly writes already-serialized buffers

3. Adjusted gzip parallel writer behavior
   - tuned chunk size and in-flight compression depth
   - capped compression threads more conservatively
   - accounted for paired-end output so compression threads are not naively
     multiplied per output path

Files touched in this phase:

Phase 8: Compression backend experiments
----------------------------------------

Once the structural and scheduling bottlenecks had been reduced, the session
continued one level deeper and explicitly examined whether the gzip backend
itself had become a new performance factor.

Key finding:

- after the earlier pipeline work, input/output compression implementation
  choice could noticeably affect wall time

Experiment path:

1. Verified that the baseline snapshot and current code used `flate2` for gzip.

2. Switched `flate2` from the default Rust backend to `zlib-rs`.

3. Rebuilt and benchmarked again.

Observed effect:

- multi-threaded wall time improved substantially
- but output `.gz` files became much larger
- this made it clear that faster wall time at this stage could come with a
  compression-ratio tradeoff rather than representing a universally better
  implementation

Additional experiment:

- `zlib-ng-compat` was also tried as a possible compromise
- it did not deliver a better overall tradeoff in this local testing

Final decision for the codebase at the end of the session:

- keep the faster `zlib-rs` direction
- explicitly treat this as a second-stage implementation choice and not as the
  original baseline root cause

This later became part of the updated `llm-benchmark/test02` grading rubric:
strong answers should be rewarded if they can identify compression backend
choice as a second-stage factor, but they should still prioritize the baseline
structural issues first.

distinct layers:

1. Baseline structural bottlenecks
   - `read_all()`

2. Post-structure throughput work

   - compression thread budgeting

3. Second-stage implementation tuning
